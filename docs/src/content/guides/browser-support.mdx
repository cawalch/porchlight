# Browser Support

Porchlight is a **modern CSS-first framework**. It does not polyfill old
browsers. Instead, it uses a tiered strategy: required features for the
core, progressive enhancement for newer capabilities, and `@supports`
gating for experimental ones.

## Support matrix

| Tier | Meaning | Strategy |
|------|---------|----------|
| **Required** | The framework assumes this works. | No fallback. Breaks gracefully if unsupported (styles won't apply). |
| **Baseline-modern** | Safe in all current-generation browsers (Baseline 2025). | Used directly. Part of the framework's baseline. |
| **Progressive enhancement** | Works without it, improves with it. | `@supports`-gated. Component degrades to a simpler but fully functional state. |
| **Experimental** | Demo-quality, not production-ready everywhere. | `@supports`-gated with obvious fallback (static layout). |

## Required features

These are foundational CSS capabilities. Any browser from 2024+ supports
all of them:

| Feature | Baseline | Used for |
|---------|----------|----------|
| Cascade layers (`@layer`) | 2024 | The entire framework architecture |
| Custom properties (`--var`) | 2023 | All tokens |
| `:has()`, `:is()`, `:where()` | 2024 | Selectors throughout |
| `oklch()` color | 2023 | All color tokens |
| Logical properties (`inline-size`, `block-start`, etc.) | 2023 | RTL-aware layout |
| CSS nesting | 2024 | Source authoring |

## Baseline-modern features

These are safe to use in production as of mid-2026. All major browsers
(Chrome, Edge, Firefox, Safari) support them:

| Feature | Baseline | Used for |
|---------|----------|----------|
| `@scope` | 2025 | All 16 components (scoped styles without BEM) |
| `light-dark()` | 2024 | Automatic light/dark color pairs |
| `color-mix()` | 2024 | Derived tints (alert backgrounds, hover states) |
| Popover API (`popover` attribute) | 2024 | `c-menu` (popover-menu) component |
| `@starting-style` | 2024 | Dialog and popover enter animations |
| `transition-behavior: allow-discrete` | 2024 | Exit animations on `display: none` elements |
| `:user-invalid` / `:user-valid` | 2024 | `c-field` validation states |
| `text-wrap: balance` / `pretty` | 2024 | Heading and paragraph typography |
| Container queries (`@container`) | 2023 | Card, data-table responsive behavior |

## Progressive enhancement features

These features are gated behind `@supports`. The framework works perfectly
without them; they add polish where supported:

| Feature | Baseline | Enhancement | Fallback |
|---------|----------|-------------|----------|
| Anchor Positioning (`anchor-name`, `position-anchor`) | 2025 | `c-menu` popover is tethered to its trigger | Popover appears centered (UA default) |
| CSS Gap Decorations (`rule`, `rule-visibility-items`) | 2026 | Grid dividers via `data-dividers` | Normal grid gap (no divider line) |
| `text-box: trim-both` | 2025 | Optically-centered button/badge labels | Standard half-leading (1px visual offset) |
| Typed `attr()` (`attr(data-x type(<color>))`) | 2026 | `c-swatch` reads color from `data-color` | Inline `style` attribute |
| View Transitions | 2025 | Smooth cross-page fades on docs site | Hard page reload |
| `contrast-color()` | 2026 | Auto-contrast text on tinted backgrounds | Explicit paired tokens (current approach) |

## Experimental features

These are not yet Baseline. They are `@supports`-gated and degrade to a
static but functional state:

| Feature | Chrome | Enhancement | Fallback |
|---------|--------|-------------|----------|
| `interpolate-size: allow-keywords` | 149 | Smooth `height: auto` animations | `grid-template-rows: 0fr -> 1fr` technique |
| Scroll-driven animations (`animation-timeline: scroll()`) | 149 | Scroll-progress bar fills as you scroll | Bar hidden (no false progress) |
| Scroll-driven animations (`animation-timeline: view()`) | 149 | Elements fade in on scroll | Elements always visible |
| `field-sizing: content` | 123 | Textarea grows to fit content | Fixed height |

> **Note on `@property`-registered `contrast-color()`:** While
> `contrast-color()` became Baseline in April 2026, MDN warns that
> mid-tone backgrounds can produce unexpected results. Porchlight keeps
> explicit paired tokens (`--pl-color-accent` / `--pl-color-accent-text`)
> with WCAG-AA verified contrast, and validates them in the deterministic
> contrast test suite.

## How degradation works

Every `@supports` gate follows the same pattern:

```css
/* Base: always applies */
.c-thing {
  /* functional styles */
}

/* Enhancement: only where supported */
@supports (feature: value) {
  .c-thing {
    /* progressive polish */
  }
}
```

The base styles are always sufficient. The `@supports` block adds a
capability that improves the experience but is not required for
functionality.

## Testing

Porchlight's test suite includes:

- **85 Playwright tests** across 14 spec files
- **Contrast tests** that compute WCAG ratios from OKLCH values (not
  axe-core, which cannot parse OKLCH computed colors as of v4.12)
- **Accessibility-mode tests** for RTL, forced-colors, reduced-motion,
  and 200% zoom
- **Smoke tests** for every preview and docs page
